# Data Requirements & Transformations

Detailed rules for transforming DB2 data to PostgreSQL format.

## Data type conversions

| DB2 Type | Rule | Postgres Type |
|----------|------|---------------|
| BIGINT | Direct | BIGINT |
| INTEGER | Direct | INTEGER |
| SMALLINT | Direct | SMALLINT |
| DECIMAL(p,s) | Numeric amounts → NUMERIC; amounts in µL → INTEGER (round/truncate) | NUMERIC or INTEGER |
| VARCHAR(n) | Direct (UTF-8 check) | VARCHAR(n) |
| CHAR(n) | Trim trailing spaces | VARCHAR(n) or ENUM-backed VARCHAR |
| DATE | Direct | DATE |
| TIME | Direct | TIME |
| TIMESTAMP | Direct | TIMESTAMP |
| BLOB/CLOB | If needed, base64 encode | BYTEA or TEXT |

## Business logic transformations

### Amounts (SAMPLE.amount)
- DB2: DECIMAL(10,2) with unit unclear
- Target: INTEGER in microliters
- Rule: Determine unit in source, convert to µL, round to nearest integer. Document conversion factor.

### Status values
- DB2: CHAR(32) with values like `'AVAILABLE'`, `'USED'`, `'NOT_AVAILABLE'`, etc.
- Target: VARCHAR(32) with enum check; values must match Java enum:
  - `PENDING`
  - `AVAILABLE`
  - `NOT_AVAILABLE`
- Rule: Map DB2 status → Postgres enum value. Log unmapped values as errors.

### IDs & sequences
- DB2: IDs generated by triggers or max-ID logic
- Target: PostgreSQL sequences with `@GeneratedValue(strategy = SEQUENCE)`
- Rule: Preserve existing IDs (do not regenerate), but ensure no gaps or conflicts. Reset sequence `nextval` to `MAX(id) + 1`.

### Audit columns
- Source: May not exist in DB2
- Target: `userstamp`, `created`, `version` required
- Rule:
  - `userstamp`: Backfill with a default user (e.g., `'MIGRATION'`) or extract from DB2 if available
  - `created`: Backfill with current timestamp (or extract from DB2 if available)
  - `version`: Backfill with `1` for all rows (first version after migration)

### Location data
- DB2: May use `location_id` FK or similar
- Target: `container_id` + `placecode` pair (see `sample-service` ADR 0001)
- Rule: Map location_id → (container_id, placecode) via a lookup table. Validate all positions exist in target CONTAINER_LOCATION grid.

## Validation rules

### Row count validation
After each table load, verify:
```
COUNT(*) in DB2 = COUNT(*) in Postgres
```

### Checksum validation
For tables without LOBs, compute a checksum (CRC or hash of all columns) and compare source ↔ target.

### Constraint validation
After load, run:
```sql
-- Check foreign keys
SELECT * FROM <table> WHERE <fk_col> NOT IN (SELECT id FROM <referenced_table>);

-- Check unique constraints
SELECT <uk_cols>, COUNT(*) as cnt FROM <table> GROUP BY <uk_cols> HAVING COUNT(*) > 1;

-- Check not-null constraints
SELECT * FROM <table> WHERE <not_null_col> IS NULL;
```

### Sample data spot-check
Extract 100 random rows from each table in DB2 and Postgres, compare field-by-field.

## Null handling

- DB2 NULL → Postgres NULL (no change)
- Empty strings (`''`) in VARCHAR:
  - If DB2 treats `''` as NULL, convert to NULL in Postgres
  - Otherwise, preserve as empty string
- Document per column if DB2 schema is ambiguous

## Character encoding

- Assume DB2 data is ISO-8859-1 or EBCDIC; re-encode to UTF-8 for Postgres
- Test on a sample first (run through iconv or Python's `decode`/`encode`)
- Flag any non-ASCII characters that don't survive re-encoding

## Timing & sequencing

1. Extract all tables in order of no-FK-dependencies first
2. Transform in any order (independent)
3. Load in order of FK dependencies (bottom-up):
   - CV tables first (controlled vocabulary)
   - Base tables (SAMPLE, CONTAINER, etc.)
   - Tables with FK to base tables

## Cleanup post-load

After all data is loaded and validated:
- Drop any temporary tables or working tables used during migration
- Ensure all Postgres audit triggers are installed (run `bcapp.enable_audit_for_schema('sample')`)
- Test one INSERT/UPDATE on each table to confirm triggers fire correctly

## Success criteria

- [ ] Row counts match for all tables
- [ ] No FK violations
- [ ] No unique constraint violations
- [ ] No null violations on NOT NULL columns
- [ ] 100% sample spot-check passed
- [ ] Audit triggers capture changes
- [ ] Optimistic locking (version column) works on test writes
- [ ] sample-service boots and runs a read query successfully
